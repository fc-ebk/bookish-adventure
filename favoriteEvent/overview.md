# 찜 정모 기능 Overview (단계별 전략)

## 목적

정모를 모임보다 앞선 **최소 관심 단위**로 정의하고
관심 → 재확인 → 참여/모임 가입으로 이어지는 흐름을 단계적으로 만든다.

이 과정에서 수집되는 데이터는 **개인 추천 및 넛지 고도화의 기반**으로 활용한다.

---

## 전체 구조 요약

* **1차 (MVP)**: 찜 정모가 실제 행동을 만드는지 검증
* **2차**: 찜한 정모가 마감돼도 흐름이 끊기지 않도록 루프 연결
* **3차**: 충분히 쌓인 관심 데이터를 기반으로 넛지 강화
* **기타 논의**: 이후 확장 가능성을 열어두는 항목들

---

## 1차: MVP — 찜 정모 기본 검증

### 핵심 목표

* 정모 찜이 참여/모임 가입 행동으로 이어지는지 확인
* 최소 기능으로 빠르게 개발·운영·지표 확인

### 주요 기능

* 정모 찜 / 해제
* 찜한 정모 리스트

  * 진행 예정 정모만 노출
  * 정렬: 오늘/내일, 이후 일정
  * 기존 정모 카드의 압축 UI
* 알림

  * 조건: 마감된 정모에 자리 발생 시에만 발송

### 데이터 수집

* BigQuery 기반 행동 로그

  * 찜/해제
  * 참여 실행
  * 모임 가입 실행
  * 알림 유입 여부(source)

---

## 2차: 루프 확장 — 마감 이후 연결

### 문제

* 찜한 정모가 마감되면 사용자의 흐름이 끊김

### 접근

* 마감된 정모 이후 추천 우선순위

  1. 같은 모임의 다른 진행중/예정 정모
  2. 해당 모임에 정모가 없을 경우에만 유사 정모
* 노출 위치

  * 찜 리스트의 적절한 곳
  * 홈 화면 개인화 섹션

### 목적

* 마감을 실패 경험으로 두지 않고 다음 기회로 전환
* 정모 단위 관심을 모임 단위 관계로 확장

---

## 3차: 넛지 강화 — 관심 기반 배너 전략

### 전제

* 찜, 참여, 가입 데이터가 충분히 누적된 상태
* 관심은 있으나 행동하지 않은 사용자 구분 가능

### 전략

* 강한 넛지 배너는 **관심이 검증된 사용자에게만** 노출
* 무차별 노출이 아닌 조건 기반 노출

  * 반복적으로 찜했으나 참여/가입하지 않은 경우
  * 특정 모임/유형에 관심이 누적된 경우
* 위치

  * 홈 화면 개인화 영역
  * 앱 실행시 하단/중앙/상단 배너
* 목적

  * 압박이 아닌 맥락 상기
  * 찜 데이터를 행동 유도 자산으로 전환

---

## 기타 논의 (후속 고려 사항)

### UX 라이팅 통일

* 찜, 참여, 가입, 알림 관련 표현을 서비스 전반에서 일관되게 정리
* 기능 이해 비용 감소, 신뢰도 유지 목적

### 정모 인지 데이터 수집

* 사용자가 정모를 어디서 인식했는지 기록

  * 홈 / 모임 / 찜 리스트 / 알림
* 찜 발생 맥락 및 추천 품질 개선에 활용

### 알림 효용 추적

* 1차에서는 구현하지 않음
* 알림 트리거 구조만 확장 가능하게 설계
* 알림 종류가 늘어나는 시점에 본격 도입

### 시스템 알림 타이밍 조정

* 시스템 알림이 뜨는 시점 변경
* 시스템 알림을 조금 더 매력적으로 바꾸기
  * 디자인
  * 텍스트 변경(UX 라이팅과 비슷한 맥락)

---

## 최종 요약

1차에서는 찜 정모의 효과를 검증하고,
2차에서는 마감 이후에도 흐름을 이어주며,
3차에서는 누적된 관심 데이터를 기반으로 넛지를 강화한다.
모든 단계의 데이터는 장기적으로 개인 추천과 행동 유도 고도화에 활용된다.
